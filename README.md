# PII-Safe-Gateway

Reverse-proxy для корпоративного доступа к внешним LLM (OpenAI-совместимый API),
который перед отправкой запроса **автоматически очищает вложенные документы
от персональных данных** (PII) с помощью локальной LLM, работающей полностью
на стороне компании — данные физических лиц никогда не покидают периметр
в исходном виде.

> **Статус проекта:** Filter Service (локальный NER-движок на LLM) полностью
> реализован и протестирован. Gateway (проксирование `/v1/chat/completions`,
> извлечение текста из файлов, безопасная замена PII на маркеры) — в разработке.

---

## Идея и архитектура

```
┌──────────────┐        ┌─────────────────────────┐       ┌──────────────────┐
│   Клиент     │───────▶│   PII Gateway (FastAPI)   │──────▶│  Внешний LLM API │
│ (OpenAI SDK) │        │  порт :8000                │       │  (OpenAI/др.)     │
└──────────────┘        └───────────┬────────────────┘       └──────────────────┘
                                     │ (если есть файл-вложение)
                                     ▼
                         ┌─────────────────────────┐
                         │  Filter Service            │
                         │  (llama.cpp + FastAPI)     │
                         │  порт :8001                 │
                         └─────────────────────────┘
```

**Ключевой архитектурный принцип — NER + Fuzzy Matching, а не переписывание
текста целиком:**

1. Gateway извлекает текст из TXT/DOCX/PDF, режет на чанки, быстрыми regex
   находит *кандидатов* на PII (email, телефоны, длинные числа, даты).
2. Filter Service (локальная LLM, Qwen2.5-7B-Instruct) получает текст и
   кандидатов, **семантически** решает, что из этого реальный PII конкретного
   физлица, а что — нет (дата документа vs дата рождения, ИНН физлица vs
   номер договора и т.п.), и возвращает структурированный JSON с точными
   координатами найденных сущностей.
3. Gateway **хирургически** заменяет найденные фрагменты на типизированные
   маркеры (`[PERSON]`, `[INN]`, `[DATE_OF_BIRTH]`...) прямо по индексам в
   исходной строке — форматирование документа не страдает.
4. Очищенный текст подставляется в `messages[]` и запрос уходит во внешний
   LLM как обычно.

Почему не "попроси LLM переписать текст без PII" — потому что при таком
подходе невозможно гарантировать сохранение структуры документа (таблицы,
нумерация, форматирование), а также сложно защититься от того, что модель
случайно "пересочинит" часть документа. NER-подход даёт точечные, проверяемые
изменения.

## Быстрый старт

### 0. Требования

- Docker Engine 24+, Docker Compose v2 (плагин)
- ~15 ГБ свободного места (сборка + веса модели)
- Рекомендуемое железо: 8+ ядер CPU, 16+ ГБ RAM (тестировалось на Ryzen 7 8845H, 32 ГБ DDR5)

### 1. Клонирование и конфигурация

```bash
git clone https://github.com/<ваш-username>/pii-safe-gateway.git
cd pii-safe-gateway
cp .env.example .env
```

Откройте `.env` и при необходимости поправьте под своё окружение (порты,
число потоков CPU — см. таблицу параметров ниже).

### 2. Скачивание модели

```bash
mkdir -p models
pip install -U "huggingface_hub[cli]"
huggingface-cli download bartowski/Qwen2.5-7B-Instruct-GGUF \
    Qwen2.5-7B-Instruct-Q4_K_M.gguf \
    --local-dir ./models --local-dir-use-symlinks False
mv models/Qwen2.5-7B-Instruct-Q4_K_M.gguf models/qwen2.5-7b-instruct-q4_k_m.gguf
```

### 3. Запуск Filter Service (CPU-профиль по умолчанию)

```bash
docker compose --profile cpu build filter-service-cpu
docker compose --profile cpu up filter-service-cpu
```

Дождитесь в логах:
```
Self-test пройден (негативный + позитивный + hint-rejection + combined-regression): ...
Filter Service готов к работе.
```

### 4. Проверка

```bash
curl http://localhost:8001/health
```

Swagger UI: **http://localhost:8001/docs** → `POST /extract-pii`.

Пример запроса:
```json
{
  "text": "Иванов Иван Иванович, тел. +7 921 555-12-34, дата рождения 12.05.1990.",
  "hints": []
}
```

Ожидаемый ответ — три сущности типов `PERSON`, `PHONE`, `DATE_OF_BIRTH`.

### 5. Включение GPU-профиля (Radeon и другие AMD iGPU через Vulkan)

```bash
docker compose --profile gpu build filter-service-gpu
docker compose --profile gpu up filter-service-gpu
```

В `.env` установите `N_GPU_LAYERS` (например, 30) — количество слоёв модели,
выгружаемых на GPU. `0` = чистый CPU.

| `EXTERNAL_LLM_BASE_URL` / `EXTERNAL_LLM_API_KEY` | адрес и ключ внешнего OpenAI-совместимого API (для Gateway) | — |

## Инженерные находки в процессе разработки

Ниже — сжатый журнал нетривиальных проблем, найденных и решённых при
реализации Filter Service. Раздел сохранён намеренно — это часть учебной
ценности проекта.

### 1. GBNF-грамматика llama.cpp: подводные камни синтаксиса

- Имена правил допускают только буквы, цифры и дефис `-`; символ `_` тихо
  обрывает парсинг имени правила без исключения в Python — ошибка видна
  только в stderr C++-парсера.
- Тело правила не может переноситься на новую строку при скобочной глубине 0
  — иначе парсер решает, что правило завершено. Длинные правила надёжнее
  писать одной строкой.
- **Важный урок:** `LlamaGrammar.from_string()` не бросает исключение на
  синтаксически неверной грамматике — она просто перестаёт ограничивать
  вывод модели. Это требует **активного self-test** при старте сервиса
  (позитивный + негативный сценарий), иначе баг остаётся незамеченным и
  единственным сигналом становится тихая деградация качества в проде.

### 2. Порядок полей в JSON-схеме — не косметика

Изначально схема была `start, end, text, type, reason`. Из-за авторегрессивной
природы генерации модель успевала "закоммититься" к типу сущности **до**
того, как сформулировать обоснование — в результате `reason` превращался в
постфактум-оправдание уже принятого (иногда неверного) решения. Смена порядка
на `start, end, text, reason, type` заставила модель реально рассуждать
**перед** вердиктом (chain-of-thought внутри структурированного вывода) —
это напрямую устранило ошибки вида "дата документа помечена как дата рождения".

### 3. Явный вердикт `NOT_PII` вместо "молчаливого исключения"

Просьба в промпте "просто не включай нерелевантного кандидата в ответ"
провоцировала модель на побочный эффект: не желая "потерять" кандидата
незамеченным, модель включала его с типом `OTHER` вместо полного исключения
(например, номер договора помечался как `OTHER` вместо того, чтобы быть
отклонённым). Решение — сделать отклонение кандидата **обязательным
позитивным действием**: для каждого regex-кандидата модель обязана вернуть
ровно один объект, либо с реальным типом, либо с явным типом `NOT_PII`.

### 4. GBNF grammar на моделях с большим словарём — дорогая операция

При активной constrained-генерации llama.cpp проверяет допустимость **каждого
токена словаря** на **каждом шаге** генерации. У Qwen2.5 словарь ~152 000
токенов (мультиязычный) — против ~32 000 у Llama/Mistral. Эмпирически это
давало устойчивые **15–27 секунд** на короткий чанк текста (несколько
десятков сущностей эквивалент), что превышает целевые показатели проекта.

Отключение grammar в пользу сильного few-shot промпта + мягкого
JSON-парсинга (`json_repair` как сеть безопасности) не устранило проблему
полностью (текущее время ~15–21 сек на CPU), что указывает на более общее
ограничение — стоимость авторегрессивной генерации 7B-модели в Q4-квантовании
на CPU для ответа из нескольких сущностей с текстовым обоснованием каждой.

**Мы сознательно отдали приоритет качеству перед скоростью**, как это прямо
допускалось требованиями проекта: контекстная точность (дата рождения vs
дата документа, ИНН физлица vs номер договора) подтверждена серией
регрессионных тестов и работает стабильно. Пути дальнейшей оптимизации
скорости, не затронутые в рамках текущей итерации:

- **GPU-профиль** (Vulkan, Radeon 780M) — наиболее вероятный кандидат для
  кардинального ускорения без потери качества;
- переход на модель с меньшим словарём (например, Phi-3.5-mini, ~32k
  токенов) при сохранении русскоязычного качества — требует отдельной
  оценки;
- сокращение длины генерируемого `reason` до 1-2 слов ценой части
  интерпретируемости решений (риск повторения регрессии из п.2);
- батчинг нескольких hint-кандидатов в параллельные независимые запросы
  вместо одного последовательного.

## Тестирование Filter Service

Полный набор регрессионных сценариев, подтверждающих контекстную точность
(ключевую ценность проекта), встроен как **self-test при старте контейнера**
— сервис не поднимется, если хотя бы один сценарий не проходит:

1. Негативный — текст без PII → пустой список сущностей.
2. Позитивный — очевидное ФИО+телефон → находятся оба.
3. Hint-rejection — номер договора среди кандидатов → полностью отклонён.
4. Combined-regression — номер договора и ИНН физлица в одном тексте →
   находится только ИНН, номер договора отклонён (регрессионный тест на
   реальный баг, обнаруженный в процессе разработки).

Ручное тестирование — через Swagger UI (`/docs`) на `POST /extract-pii`,
примеры запросов приведены в разделе "Быстрый старт".

## Потенциальное развитие проекта

- **Reverse-mapping (деанонимизация ответа)** — восстановление реальных
  значений PII в ответе внешней LLM, если она процитирует маркер. Требует
  хранения соответствия `маркер → оригинал` в рамках запроса/сессии
  (in-memory или Redis). Сознательно не реализовано в текущей версии —
  Gateway выполняет только анонимизацию входящих данных.
- **OCR для сканированных документов** — текущая реализация работает только
  с текстовым слоем PDF/DOCX.
- **Кэширование результатов NER** для повторяющихся фрагментов документов.
- Переход на модель с меньшим словарём или квантование меньшего размера
  для соответствия целевым 2 секундам на CPU без потери контекстной точности.
